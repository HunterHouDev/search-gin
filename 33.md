## P0 — 严重缺陷

### P0-1: [已修复] SSE & WS Hub `init()` 递归 goroutine 泄漏

**文件**：`internal/sse/hub.go`, `internal/ws/hub.go`  
**状态**：✅ **已修复**（`bb7a53a`）

```go
// 修复后
var hubRunning atomic.Bool

func startHub() {
    if hubRunning.CompareAndSwap(false, true) {
        go func() {
            defer hubRunning.Store(false)
            defer func() {
                if r := recover(); r != nil {
                    fmt.Printf("panic: %v\n", r)  // 不再递归重启
                }
            }()
            DefaultHub.Run()
        }()
    }
}
```

**修复方式**：`atomic.Bool` CAS 控制最多一个 goroutine 运行；panic 后只打日志不重启。

### P0-2: [已修复] `forwardRequest` body 可能读取为空

**文件**：`internal/service/remote_operation.go`  
**状态**：✅ **已修复**（`bb7a53a`）

```go
// 修复前
bodyBytes, err := io.ReadAll(c.Request.Body)

// 修复后
bodyBytes, err := c.GetRawData()
```

**修复方式**：使用 Gin 内置 `c.GetRawData()`（缓存 body，不受中间件预读影响）。

### P0-3: [已注释] `FillURLs` 流 URL host 使用 `pickLocalIP`

### P2-2: [已注释] `remoteClient`/`peerClient` 超时文档

**文件**：`internal/service/file_downloader.go`  
**状态**：📝 **已加注释说明**（`bb7a53a`）

```go
// NOTE: 两个客户端均有显式超时（peerClient 5s, remoteClient 2s），不存在"无超时永久阻塞"问题。
```

追踪发现 `remoteSearchTimeout` 定义在 `remote_search.go`（30s）。结论：超时已配置，注释补充后风险消除。

### P2-3: [未修复] 索引缓存 30s 去抖窗口

**文件**：`internal/service/index_engine_cache.go`  
**状态**：❌ **未修复**

`FlushCache` 在 `GracefulShutdown` 时调用，理论上窗口内最后 30 秒的数据可能丢失。但 `FlushCache` 会等待去抖定时器或直接写磁盘。

**建议**：减少去抖窗口到 5s，或关闭时立即写盘。

### P2-4: [未修复] `GetRefreshIndex` 同步阻塞

**文件**：`internal/handler/file_stream_controller.go:GetRefreshIndex`  
**状态**：❌ **未修复**

`GetRefreshIndex` 调 `ScanAll()` 是同步的，扫描数分钟期间 HTTP 不响应。

**建议**：改为 `go ScanAll()` 立即返回 200。

### P2-5: [未修复] 增量扫描结果不持久化到磁盘

`rebuildWithBucketIncremental` 调用 `installIndexSkipDisk`（不调 `saveIndexToCache`），增量扫描结果在 30 秒窗口内不写盘。

**建议**：增量扫描也触发 `saveIndexToCache`，或单独 5s 去抖。

### P2-6: [未修复] `LogMem.Add` 双倍 I/O

`LogMem.Add` 先写内存再调 `utils.InfoFormat` 写磁盘。内部日志双倍写入。

**建议**：`LogMem.Add` 仅写内存，由独立 goroutine 定期 flush 到磁盘。

### P2-7: [未修复] `ScanProgress.mu` 嵌套锁风险

`setPhase` 无锁（注释说由调用方持锁），但所有公开方法都持锁调用 `setPhase`——如果调用方自己持锁再调 `setPhase` 会死锁。

**建议**：`setPhase` 改为内部私有不导出，所有公开方法统一加锁。

```go
var streamSecret = []byte{

```

AES-256-GCM 密钥硬编码在源码中。虽然方便跨节点共享，但任何有仓库访问权限的人可解密所有 stream token。

### P3-12: [新发现] `stream_crypto.go` 中 `EncryptStreamToken` 未处理 `err2` 覆盖

**文件**：`internal/service/remote_search.go:258-259`

```go
imgToken, err := utils.EncryptStreamToken(imgExpire)
streamToken, err2 := utils.EncryptStreamToken(streamExpire)  // err2!
if err != nil || err2 != nil {
```

`err` 被第一个调用赋值后在第二个调用中仍在使用——两个不同变量名 `err` / `err2` 没问题，但检查时将 `err` 和 `err2` 都检查了，正确。但若第一个调用失败、第二个成功，`err` 非空导致两个 token 都为空。建议分别检查。

### P3-13: [新发现] WS Hub `GetOnlineUsers` 新增 `contains` 辅助函数

**等级**：Low
